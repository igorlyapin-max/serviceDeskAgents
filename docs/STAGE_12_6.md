# Этап 12.6: профили разрешения атрибутов

Этап 12.6 выносит логику заполнения слотов в отдельный каталог профилей разрешения атрибутов. Схема слотов выбирает готовое правило заполнения, а профиль описывает ordered pipeline: извлечение из текста, чтение из кейса, вызовы tools, поиск по истории, уточняющие вопросы и fallback.

## Реализовано

- Новый контракт `contracts/config/attribute-resolution-profiles.schema.json`.
- Новый домен Config Registry `attribute_resolution_profiles`.
- Default-профили:
  - `Поиск логина в AD по ФИО`;
  - `Устройство пользователя из AD`;
  - `Устройство из CMDB`;
  - `Подсеть по локации из CMDB`;
  - `Пользователь запроса доступа из AD`;
  - `История успешных сбросов пароля`.
- Read-only tool `search_ad_users` и endpoint profiles `mock.identity` / `n8n.identity`.
- Раздел Admin UI `Сценарии обработки -> 0. Разрешение атрибутов`.
- CRUD-операции профилей через `draft/validate/regression/activate`.
- Шаги профиля редактируются карточками формы, без JSON-поля и без сводной таблицы профилей.
- Редактор слотов использует `Способ заполнения` и `Профиль разрешения атрибута` вместо прямого выбора технического источника.
- Dry-run сценария возвращает `attribute_resolution` со списком шагов профиля.
- Операторская панель показывает способ заполнения и шаги разрешения атрибутов в шаге `1. Приём и нормализация`.

Дополнение этапа 12.7 развивает эту модель: профиль разрешения атрибута трактуется как mini-workflow одного слота с промежуточными атрибутами, переходом при неоднозначности, отдельными confidence thresholds и пакетом передачи Л1.

## Цель

- Убрать hardcoded список источников слота из UI и backend.
- Заменить поле `source` в слоте на выбор способа заполнения и профиля разрешения атрибута.
- Дать администратору отдельное меню для создания, модификации и удаления профилей.
- Оставить tools runtime-сущностью исполнения, а не основной сущностью настройки слота.

## Раздел UI

- Пункт меню: `Сценарии обработки -> 0. Разрешение атрибутов`.
- Операции:
  - `Создать`;
  - `Модифицировать`;
  - `Удалить`.
- UI-обвязка:
  - выпадающий выбор профиля;
  - форма создания;
  - форма модификации;
  - карточки шагов профиля с отдельными полями для tool, условий, уточняющих вопросов, переходов и фильтра истории заявок;
  - подтверждение удаления;
  - сохранение через `draft/validate/regression/activate`;
  - audit event для каждой операции.

## Профиль разрешения атрибута

Профиль содержит:

- название;
- статус;
- целевой слот;
- входные слоты;
- выходные слоты;
- упорядоченные шаги;
- условия переходов;
- правила уточняющих вопросов;
- лимит попыток;
- confidence threshold;
- fallback;
- audit/log флаги.

Типы шагов:

- `llm_extract`;
- `case_read`;
- `tool_call`;
- `ticket_history_search`;
- `condition`;
- `clarification`;
- `fill_slot`;
- `escalate`.

## Изменение сценариев

В редакторе слота остается:

- `Способ заполнения`:
  - вопрос пользователю;
  - текущий кейс;
  - извлечение моделью;
  - профиль разрешения атрибута;
  - ручное заполнение оператором.
- `Профиль разрешения атрибута`, если выбран профильный способ.

Для выбранного профиля UI показывает только read-only сводку: входы, выходы, fallback, audit и возможные уточняющие вопросы.

## Пример

Профиль `Поиск логина в AD по ФИО`:

1. Извлечь ФИО из текста обращения через LLM.
2. Найти пользователя в AD.
3. Если найден один пользователь, заполнить `login_name`.
4. Если найдено несколько пользователей, запросить должность, подразделение или табельный номер.
5. Повторить поиск с уточнением.
6. Если результат неоднозначен после лимита попыток, передать оператору.

## Валидации

- Слот может ссылаться только на существующий профиль, который выдает этот слот как output.
- Профиль может объявлять дополнительные `output_slots`, которые не входят в конкретную схему слотов; такие выходы считаются возможностями профиля и не блокируют сохранение схемы.
- Tool step ссылается на существующий tool, endpoint profile и operation.
- Каждый step использует только объявленные входы и выходы.
- Clarification step содержит вопрос и уточняемые поля.
- History step содержит фильтры статусов, качества, периода и разрешенных полей.
- Fallback задан для неоднозначного и неуспешного пути.
- Слот не может ссылаться на профиль, не выдающий именно этот слот как output.

## Критерии выхода

- В админской консоли есть раздел `0. Разрешение атрибутов` с CRUD.
- В разделе нет сводки профилей и JSON-редактора шагов; порядок поведения профиля настраивается через карточки.
- В сценариях слот выбирает готовый профиль, а не технический источник данных.
- Pipeline поиска `login_name` для сброса пароля описывается конфигурацией.
- История заявок используется только через явно настроенный профиль.
- Тестовый прогон показывает шаги разрешения атрибутов и вопросы уточнения без реального вызова внешних систем.
