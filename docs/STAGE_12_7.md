# Этап 12.7: разрешение слота как mini-workflow

Этап 12.7 уточняет модель `0. Разрешение атрибутов`: профиль больше не считается простым списком источников. Профиль является управляемым mini-workflow для заполнения одного слота, с промежуточными атрибутами, ветвлениями, неоднозначностью, повторными попытками и передачей Л1.

## Реализовано

- Контракт `attribute_resolution_profiles` расширен совместимыми полями:
  - `resolution_mode`;
  - `attempt_scope`;
  - `intermediate_attributes`;
  - `confidence_thresholds`;
  - `ambiguity_policy`;
  - `operator_handoff_package`;
  - `on_ambiguous_step` и `ambiguity_condition` в шагах.
- Добавлены типы шагов:
  - `rag_search`;
  - `operator_handoff`.
- Default-профиль `Поиск логина в AD по ФИО` переработан как ветвящийся mini-workflow:
  1. извлечь логин, ФИО и контакты из текста;
  2. проверить подсказки по похожим заявкам/RAG;
  3. проверить найденный логин в AD;
  4. при необходимости искать пользователя по ФИО;
  5. проверить единственное совпадение;
  6. при неоднозначности запросить должность, подразделение или табельный номер;
  7. заполнить `user_login` и дополнительный `user_id`;
  8. при неуспехе передать Л1 с пакетом контекста.
- Backend-валидация проверяет:
  - переходы `on_success_step`, `on_failure_step`, `on_ambiguous_step`;
  - объявление промежуточных атрибутов;
  - корректность `from_attribute`;
  - пакет передачи Л1;
  - порядок порогов confidence.
- Dry-run сценария возвращает `resolution_state` по слотам:
  - текущий шаг;
  - выполненные шаги;
  - номер попытки;
  - следующий вопрос;
  - признаки неоднозначности;
  - пакет передачи Л1.
- Admin UI показывает:
  - режим выполнения профиля;
  - область действия лимита попыток;
  - промежуточные атрибуты;
  - отдельные пороги confidence;
  - политику неоднозначности;
  - переход при неоднозначности на уровне шага.
- Operator UI в шаге `1. Приём и нормализация` показывает текущий mini-workflow разрешения слота, вопрос, прогресс dry-run и пакет Л1.

## Архитектурное решение

Слот остается бизнес-атрибутом сценария. Он не знает, откуда именно берется значение. Способ получения значения описывает профиль разрешения атрибута.

Промежуточные значения вроде `last_name`, `department`, `employee_number`, `ad_candidates` не обязаны быть слотами сценария. Они являются внутренним контекстом профиля и могут попасть в пакет передачи Л1 или быть использованы для следующей попытки поиска.

Инструменты остаются runtime-адаптерами. В профиле администратор настраивает шаг "проверить пользователя в AD", а технически этот шаг ссылается на tool/endpoint/operation.

## URL

- Интерфейс администратора: `http://127.0.0.1:18088/admin`.
- Панель оператора: `http://127.0.0.1:18088/operator`.
- Раздел администратора: `Сценарии обработки -> 0. Разрешение атрибутов`.

## Критерии выхода

- Профиль разрешения атрибута управляется как mini-workflow, а не как один источник данных.
- Неоднозначность отделена от технической ошибки.
- Оператор видит, почему система задает вопрос по слоту и что будет после ответа.
- Переход Л1 получает полный пакет контекста по разрешению слота.
- Smoke-проверка подтверждает контракт, dry-run и UI-маркеры нового поведения.
