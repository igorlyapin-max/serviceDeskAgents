# Этап 12.7: resolver-профиль разрешения атрибута

Этап 12.7 упрощает `1. Разрешение атрибутов`: профиль больше не является mini-workflow с ручными переходами. Профиль описывает детерминированный resolver:

```text
признаки -> операция разрешения -> оценка результата операции -> матрица решений -> заполнить слот / уточнить / передать человеку
```

## Реализовано

- Контракт `attribute_resolution_profiles` переведен на модель:
  - `input_attributes`;
  - `candidate_source`;
  - `result_policy`;
  - `decision_policy`;
  - `clarification_policy`;
  - `handoff_policy`.
- Из модели и UI убраны workflow-поля профиля:
  - `steps`;
  - `resolution_mode`;
  - `attempt_scope`;
  - `intermediate_attributes`;
  - `ambiguity_policy`;
  - `operator_handoff_package`;
  - `on_success_step`, `on_failure_step`, `on_ambiguous_step`;
  - ручные условия вида `ad_candidates.count == 1`.
- Default-профиль `Поиск логина в AD по ФИО` описывает:
  - признаки из текста и уточнений;
  - операцию разрешения `search_ad_users`;
  - чтение ответа AD через `result_policy`;
  - матрицу решений для пустого, одного и нескольких результатов;
  - вопрос при неоднозначности;
  - пакет передачи человеку.
- Backend dry-run применяет матрицу решений и возвращает `resolution_state`:
  - операцию разрешения;
  - сводку результата операции;
  - принятое решение;
  - вопрос;
  - пакет передачи человеку.
- Admin UI показывает форму без JSON-редактора и без workflow-шагов:
  - признаки для поиска;
  - операцию разрешения атрибута;
  - как оценивать результат операции;
  - матрицу решений;
  - уточнение и передачу человеку.
- Operator UI в шаге `1. Приём и нормализация` показывает операцию разрешения, решение dry-run, вопрос и пакет передачи.

## Архитектурное решение

LLM может извлекать признаки из текста и помогать формулировать вопрос, но не управляет ветвлениями. Решение принимает Python-бэкенд по `decision_policy`.

Формальный язык условий в UI не вводится. Вместо этого используется фиксированная матрица:

- результата нет;
- результат один;
- результатов несколько;
- источник недоступен;
- лимит попыток исчерпан.

Для каждого случая администратор выбирает действие из ограниченного списка: заполнить при достаточной уверенности, уточнить, передать человеку, эскалировать, оставить пустым или остановить в отладке.

## URL

- Раздел администратора: `Сценарии обработки -> 1. Разрешение атрибутов`.

## Критерии выхода

- Профиль разрешения атрибута управляется как resolver, а не как workflow.
- Администратор не пишет условия и переходы вручную.
- Dry-run показывает, почему слот заполнен, почему задан вопрос или почему нужна передача человеку.
- Smoke-проверка подтверждает контракт, dry-run и UI-маркеры нового поведения.
