# Этап 7: загрузка базы знаний и минимальный RAG

Этап 7 вводит слой загрузки базы знаний, а не просто локальную папку `knowledge/`. Workflow должен искать данные в заранее подготовленном индексе во время анализа заявки. `/tickets/analyze` не должен синхронно загружать произвольные страницы wiki или API.

## Базовые решения

- Источники базы знаний описываются в каталоге и читаются через коннекторы источников.
- `knowledge/` является локальным источником для разработки, а не единственным типом источника.
- Корпоративная wiki, ServiceDesk KB, экспорты CMDB, Git-репозитории и будущие API-источники нормализуются к единому формату документа.
- Перестроение базы знаний является явным действием оператора.
- Workflow анализа использует последний успешно построенный индекс.
- Если поиск по базе знаний падает, анализ заявки деградирует к существующим веткам ответа, уточнения, эскалации или согласования.

## Типы источников

Начальная архитектура поддерживает:

- `local_files` для локальных документов `.md` и `.txt`.
- `wiki` как заготовку коннектора корпоративной wiki.
- `servicedesk_kb` как будущий коннектор базы знаний ServiceDesk.
- `cmdb_export` как будущий коннектор структурированных экспортов.
- `git_repository` как будущий коннектор ранбуков и документации.
- `api_pull` как будущий универсальный API-коннектор.

Этап 7 реализует `local_files` и добавляет отключенные, готовые к настройке заготовки удаленных источников.

## Нормализованный документ

Все коннекторы должны производить нормализованные документы:

- `document_id`
- `source_id`
- `source_type`
- `title`
- `uri`
- `content`
- `content_version` или checksum
- `updated_at`
- metadata: service, environment, labels, owner team, security labels

Поиск и workflow должны работать с нормализованными документами и фрагментами, а не с payload конкретного коннектора.

## Действие оператора: перестроение

Перестроение базы знаний запускается оператором:

1. Оператор запускает перестроение через CLI или API.
2. Backend загружает включенные источники из каталога источников.
3. Коннекторы синхронизируют документы.
4. Документы нормализуются, разбиваются на фрагменты и индексируются.
5. Backend пишет manifest индекса со статусом, счетчиками источников, документов, фрагментов, ошибками и временными метками.
6. `/tickets/analyze` использует последний успешный индекс.

Команда перестроения не запускается автоматически на каждый запрос анализа. Управление из UI относится к этапу 8, но backend-команда, API и результат аудита относятся к этапу 7.

## Backend поиска

Для разработки без GPU этап 7 начинает с локального лексического поиска и файлового индекса в `state/`.

pgvector остается целевым production-backend, но должен находиться за тем же интерфейсом поиска. Workflow API и формат ссылок на источники не должны меняться при добавлении pgvector.

## Команды

```bash
KNOWLEDGE_REBUILD_OPERATOR=operator-1 ./scripts/rebuild-knowledge.sh
```

```bash

```

```bash

```

## Критерии выхода

- Несколько типов источников можно представить в каталоге источников.
- Локальные файлы загружаются через тот же интерфейс коннектора, что и будущие удаленные источники.
- Отключенные заготовки удаленных источников не ломают перестроение или анализ.
- Запущенное оператором перестроение создает валидированный manifest и индекс.
- Анализ заявки может ссылаться на найденные фрагменты.
- Ошибки поиска не ломают workflow.
- Дизайн позволяет добавить синхронизацию корпоративной wiki без изменения формы ответа `/tickets/analyze`.
