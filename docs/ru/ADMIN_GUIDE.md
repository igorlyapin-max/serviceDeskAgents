# Инструкция администратора AI ServiceDesk

## Назначение

Консоль администратора предназначена для настройки и контроля AI ServiceDesk Platform. Она не используется для повседневной обработки обращений клиентов.

Администратор управляет:

- сценариями обработки;
- слотами и автозаполнением слотов;
- разрешением атрибутов;
- классификацией и маршрутами;
- ReAct-планированием;
- ReAct-вызовами и матрицей запуска;
- правилами решения и эскалации;
- prompt packs;
- базой знаний;
- моделями;
- интеграциями;
- потоком обработки;
- консолью отладки и dry-run симуляциями;
- аудитом, безопасностью и правами.

URL консоли администратора:

```text
http://127.0.0.1:18088/admin
```

## Роли и права

Основные роли:

- `admin` - полный доступ к административной плоскости и консоли отладки;
- `operator` - обработка обращений и согласование действий;
- `support_l1` - чтение кейсов и базовые операторские действия;
- `support_l2` - чтение, операторские действия и диагностика ReAct-вызовов;
- `readonly` - просмотр кейсов, моделей, workflow и аудита;
- `endpoint` - техническая роль для callbacks.

Для управления потоком обработки используются права:

- `processing.read` - просмотр запусков, задач, ожиданий и событий;
- `processing.manage` - безопасные команды: cancel, retry, release lease, force timeout, escalate.

## Сценарии обработки

Раздел `Сценарии обработки` задает поведение оркестратора вокруг пяти шагов:

- `0. Слоты` - структура данных, которые нужно собрать;
- `0.1 Автозаполнение слотов` - детерминированное заполнение слотов через read-only ReAct-вызовы;
- `1. Разрешение атрибутов` - последовательность обогащения контекста и LLM-правило принятия решения;
- `2. Классификация и маршрут` - правила, confidence и маршрут обращения;
- `3. ReAct-планирование` - лимиты итераций и стоп-условия;
- `4. ReAct-вызовы и матрица запуска` - какие вызовы доступны сценарию и с каким видом запуска;
- `5. Решение и эскалация` - условия закрытия, ожидания и передачи оператору;
- `6. Промпты` - prompt pack с обязательными блоками системного промпта.

Изменения сценариев выполняются через draft, validation, regression и activation. Активная конфигурация меняется только после успешной активации.

## Вызовы и интеграции

Раздел `Вызовы и интеграции` разделяет три уровня:

- `Интеграции` - технические endpoint adapters и операции endpoint;
- `ReAct-вызовы ИИ` - бизнес-доступные вызовы для оркестратора;
- `Привязка операций` - связь ReAct-вызова с endpoint operation и маппинг параметров.

LLM не вызывает внешние системы напрямую. Она может предложить структурированный ReAct-вызов, а backend применяет Tool Registry, Execution Policy и Integration Dispatcher.

## Модели

Раздел `Модели` управляет подключениями через LiteLLM.

Поддерживаемые MVP-профили:

- `vllm_cpu` - локальный vLLM CPU backend для smoke/integration проверок;
- `openai` - OpenAI API через LiteLLM;
- дополнительные LiteLLM-подключения.

Секреты не отображаются в UI. Если значение задано, UI показывает `параметр скрыт`; если отсутствует - `параметр не заполнен`.

Запись секретов через UI предназначена только для локального MVP и dev-стенда. В production Admin UI не должен изменять `.env`; секреты задаются внешней инфраструктурой.

## База знаний

Раздел `База знаний` используется для просмотра источников, статуса индекса, chunks и тестового поиска.

Перестроение базы знаний является явным действием оператора или администратора. Источники знаний могут включать wiki, markdown, PDF, DOCX, HTML, CMDB export, runbooks и FAQ.

## Поток обработки

Раздел `Поток обработки` показывает runtime-состояние асинхронной обработки:

- активные `processing_run`;
- `agent_task` с attempt, heartbeat, lease и idempotency key;
- `wait_state` для ожидания клиента, внешнего события или согласования;
- Kafka-ready `processing_outbox`;
- timeline выбранного кейса;
- runtime-граф обработки поверх графа оркестрации.

Разрешены только безопасные действия:

- отменить run;
- повторить retryable task;
- освободить stale lease;
- завершить ожидание по timeout;
- эскалировать кейс.

Прямое редактирование слотов, workflow state и context из админки запрещено.

## Консоль отладки

Консоль отладки предназначена для проверки сценариев, мультиагентного потока и mock-перехода от тестовых интеграций к реальным endpoint-вызовам.

URL:

```text
http://127.0.0.1:18088/debug
```

Маршрут `/operator` оставлен как alias для локальной совместимости. Новое имя интерфейса - `Консоль отладки`.

Разделы консоли:

- `Одиночный прогон` - ручная проверка одного обращения, слотов, пяти шагов оркестратора и трассы тестового прогона;
- `Мультиагентный поток` - генерация набора обращений из сценариев, предпросмотр, редактирование и запуск;
- `Трассы обращений` - case-centric разбор полного цикла обработки по `case_id`;
- `Активные ожидания` - ожидания клиента, таймера, внешнего события или согласования;
- `Mock из endpoint-вызовов` - захват реальных endpoint-вызовов, обезличивание и создание mock output.

Потоки обращений генерируются от сценариев как источника истины. Если сценарий изменен, новый поток должен отражать новые слоты, ReAct-вызовы и ожидаемые ветки.

Генератор текста обращения использует `examples` слотов, затем описание сценария и инструкции извлечения. Технические expected values не вставляются в текст заявки. Они отображаются отдельно:

- `В тексте заявки` - данные, которые действительно присутствуют в сообщении клиента;
- `Ожидается из автозаполнения` - значения, которые должны прийти через ReAct/autofill;
- `Ожидается из разрешения атрибутов` - значения, которые должны быть получены через resolution profile;
- `Ожидаемый результат` - закрытие, ожидание, эскалация или out-of-scope.

После запуска консоль показывает блок `Итог работы агента`. Это бизнес-вердикт поверх технического статуса:

- `Успешно` - обязательные данные собраны, блокирующих ошибок нет;
- `Нужна проверка данных` - есть спорные данные, низкая уверенность, неоднозначность или требуется подтверждение действия;
- `Ожидает клиента` - агент задал уточняющий вопрос и должен продолжить после ответа;
- `Эскалировано` - автообработка остановлена, подготовлена передача в канал эскалации;
- `Ошибка` - сломан mock, контракт, конфигурация или выполнение.

Служебный статус `completed` означает только то, что движок завершил прогон. Для оценки результата используйте `Итог работы агента` и трассу вызовов.

В потоковом dry-run ожидаемая ветка генератора сравнивается с фактическим итогом агента. Если, например, вариант ожидал уточнение клиента, а агент завершил обращение как успешное, итог будет показан как `Нужна проверка данных`.

Нецелевые обращения должны быть близки по формату к целевому сценарию, но не соответствовать ему по смыслу. Например, финансовый запрос может содержать контактное лицо и ФИО сотрудника, но не должен запускать сценарий сброса пароля.

Импорт реальных заявок и переписок чат-бота заложен как dry-run каркас. Реальные данные должны быть помечены как `contains_real_data` и не должны запускать state-changing операции без отдельного безопасного режима.

## Mock из реальных endpoint-вызовов

Для перехода от тестовых сценариев к реальным интеграциям можно включить захват выбранной endpoint-операции.

Порядок работы:

1. Включить захват для `endpoint_id` и `operation_id`.
2. Выполнить контролируемый read-only или безопасный вызов.
3. Проверить, что response проходит `response_schema`.
4. Выполнить обезличивание captured response.
5. Создать mock example и сделать его текущим `mock_output` операции.

Captured response со статусом `not_sanitized` нельзя использовать в массовом dry-run. Если response не соответствует контракту, mock автоматически не создается: нужно обновить контракт или пометить операцию как broken.

## Наблюдаемость

Каждый HTTP-ответ получает `X-Request-ID`, а аудит сохраняет request id в деталях события. Это позволяет связывать UI-действие, backend-log и audit event.

Endpoint `/metrics` отдает Prometheus-совместимые счетчики и duration-суммы для HTTP-запросов, интеграционных вызовов и повторных callbacks. В MVP это легкий встроенный registry без внешнего collector.

Повторный callback с тем же `invocation_id` считается идемпотентным: backend возвращает ранее сохраненный результат и помечает ответ как `duplicate`.

## Аудит и безопасность

Все опасные административные действия требуют permission check и пишут audit event. Секреты должны храниться в переменных окружения или внешнем хранилище секретов; в конфигурации сохраняются только ссылки на секреты.

Перед передачей текста обращения в LLM выполняется минимальное обезличивание: email, телефоны, bearer/API tokens и присваивания вида `password=...`, `token=...`, `секрет=...`, `ключ=...` заменяются техническими маркерами. В трассе тестового прогона отображается только факт redaction и типы сработавших маркеров, а не исходные значения.

## Правила языка

UI поддерживает русский и английский языки. Технические идентификаторы не переводятся: API paths, environment variables, JSON fields, service names, Kafka topics, enum/status ids, tool names, endpoint ids и operation ids.
