

MVP AI ServiceDesk Assistant — архитектура и этапы

Цель MVP

Оператор вручную вводит обращение в AI-систему.

AI-система:

    пытается обработать обращение самостоятельно;
    если данных недостаточно — запрашивает уточнение;
    если решить задачу невозможно — формирует текст эскалации.

На первом этапе API сервисдеска отсутствует.
Оператор создает заявку вручную.


Архитектура MVP

Operator UI

   ↓

FastAPI AI Orchestrator

   ↓

LangGraph

   ├─ классификация

   ├─ RAG-поиск

   ├─ проверка полноты данных

   ├─ вызов tools

   └─ решение:

        - выполнить

        - запросить уточнение

        - эскалировать

        ↓

LiteLLM Gateway

        ↓

Ollama / vLLM


Инфраструктурное размещение

VM-1 — Linux Application Server

Назначение:

    orchestration;
    integrations;
    UI;
    integration runtime.

Состав:

VM-1 Linux

├─ React Operator UI

├─ FastAPI AI Orchestrator

├─ LangGraph

├─ Tool Registry

├─ n8n integration layer

└─ Admin/Operator API


VM-2 — Linux

Назначение:

    inference;
    RAG;
    data layer.

Состав:

VM-2 Linux

├─ LiteLLM Gateway

├─ Ollama или vLLM

├─ PostgreSQL

├─ Qdrant / pgvector

└─ Redis


LiteLLM

LiteLLM является единым LLM Gateway.

Назначение:

    единый OpenAI-compatible endpoint;
    routing моделей;
    fallback;
    abstraction layer между orchestrator и моделями.

Архитектурная роль:

FastAPI / LangGraph

        ↓

LiteLLM Gateway

        ↓

Ollama / vLLM


LangGraph

LangGraph используется как orchestration layer.

Функции:

    классификация обращений;
    stateful workflow;
    маршрутизация сценариев;
    вызов tools;
    RAG orchestration;
    решение об эскалации.


RAG

RAG используется для поиска по внутренней базе знаний.

Источники:

    wiki;
    runbooks;
    FAQ;
    инструкции;
    архитектурная документация;
    CMDB данные.

Состав:

Documents

   ↓

Chunking

   ↓

Embeddings

   ↓

Qdrant / pgvector

   ↓

Retriever

   ↓

LLM


Tools

Tools вызываются не напрямую из LLM.

Правильная схема:

LangGraph

   ↓

Tool Registry

   ↓

Tool Adapter

   ↓

External System

Примеры tools:

    check_zabbix_status;
    query_cmdb_object;
    get_service_owner;
    search_known_incidents;
    start_systemcenter_runbook.


n8n integration layer

Внешние интеграции выполняются через n8n.

Схема:

LangGraph

   ↓

Tool Registry

   ↓

n8n webhook / workflow

   ↓

External Systems

Назначение n8n integration layer:

    orchestration внешних интеграций;
    запуск workflow;
    интеграция с внешними API;
    вызов PowerShell/WinRM сценариев при необходимости;
    retry/fallback;
    аудит действий;
    нормализация ответов в JSON.

LangGraph принимает решение ЧТО вызвать.

n8n выполняет интеграционную логику КАК вызвать.


Минимальный функционал MVP

Operator UI

Функции:

    Ввод обращения;
    Просмотр ответа AI;
    Просмотр причины эскалации;
    Просмотр необходимых уточнений;
    Копирование текста для сервисдеска.

Поля обращения:

    пользователь;
    система/сервис;
    окружение;
    описание проблемы;
    приоритет.


Решения AI

1. AI может выполнить запрос

{

  "decision": "resolved_by_ai",

  "message": "Сервис восстановлен"

}


2. Недостаточно данных

{

  "decision": "needs_clarification",

  "missing_fields": [

    "service_name",

    "environment"

  ]

}


3. Требуется эскалация

{

  "decision": "requires_escalation",

  "summary": "Недоступность billing-api",

  "reason": "Требуется участие L2"

}
